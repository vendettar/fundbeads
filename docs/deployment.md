# Fundbeads Deployment Guide

This document describes the CI/CD architecture and operational procedures for deploying Fundbeads to **Cloudflare Pages**.

> **Current phase**: Frontend-only. No backend server is required — all processing runs in the user's browser. The deployment target is Cloudflare Pages.

---

## 1. Architecture Overview

```
GitHub (main / release branch)
        │
        ▼  git push (webhook trigger)
Cloudflare Pages (Native Git Integration)
        │
        ├── Build: pnpm --dir frontend build
        │
        ▼
Cloudflare Pages Hosting (fundbeads project)
        │
        ├── Preview Deployments  ← main branch (and all feature branches)
        └── Production Deployment  ← release branch
```

- **All logic runs in the browser** — no API server, no database, no secrets needed at runtime or deploy time.
- **Cloudflare Pages** natively connects to the GitHub repository, monitors branch updates, and automatically builds/deploys the site via Cloudflare's global edge network.
- **Zero secrets required in GitHub** — no API tokens or Account IDs are needed for GitHub Actions to push deployments.

---

## 2. One-time Setup

To connect the repository to Cloudflare Pages:

1. Log in to the [Cloudflare Dashboard](https://dash.cloudflare.com).
2. Navigate to **Workers & Pages → Create → Pages → Connect to Git**.
3. Select the `vendettar/fundbeads` repository.
4. Configure the build settings:

   | Setting | Value |
   | :--- | :--- |
   | Production branch | `release` |
   | Build command | `pnpm --dir frontend build` |
   | Build output directory | `frontend/dist` |
   | Root directory | `/` (repo root) |
   | Node.js version | `22` |

5. Click **Save and Deploy**. Cloudflare Pages will automatically trigger builds on every git push to the configured branches.

## 3. Branch & Deployment Model

| Git Branch | Environment | Trigger | URL |
| :--- | :--- | :--- | :--- |
| `main` | Preview / Pre-production | Every push or PR merge | `*.fundbeads.pages.dev` (Branch preview URL) |
| `release` | Production | Every push or PR merge | Custom domain (e.g. `fundbeads.app`) |
| Any PR | PR Preview | PR opened / updated | Temporary PR preview URL generated by CF |

### Branch Rules
- Feature development happens on `feature/*` branches.
- All features are merged into `main` via **Squash and Merge** or **Rebase and Merge** (avoid merge commits).
- Once `main` is validated in the preview environment, open a **PR (`main` → `release`)** to promote to production.
- Merging that PR triggers the automatic production deployment on Cloudflare Pages.

## 4. CI/CD Workflows

### `pr-ci.yml` — Pull Request Validation

- **Trigger**: Any PR targeting `main` or `release`.
- **Steps**: Install → Build (`tsc + vite build`) → Test (`vitest run`).
- **Purpose**: Gates all merges on a green build and full test suite to guarantee code stability before merging.

---

## 5. Standard Operating Procedures

### 5.1 Normal Feature Release

1. Create a `feature/my-feature` branch from `main`.
2. Open a **PR → `main`**. CI must pass.
3. Merge via **Squash and Merge**. Cloudflare Pages automatically detects the update on `main` and deploys a new preview version.
4. Validate changes at the preview URL.
5. Open a **PR (`main` → `release`)** and merge. Cloudflare Pages detects the update on `release` and deploys to **production**.

### 5.2 Emergency Hotfix

1. Create a `hotfix/fix-name` branch from `release`.
2. Commit the fix and open a **PR → `release`**.
3. Merge. Production deploys automatically.
4. **Critical**: Cherry-pick the fix back into `main` to avoid regression:
   ```bash
   git checkout main
   git cherry-pick <hotfix-sha>
   git push
   ```

### 5.3 Manual Deploy (Retrigger)

If you need to manually trigger a deployment without making a code change:
1. Go to the [Cloudflare Dashboard](https://dash.cloudflare.com) → **Workers & Pages → fundbeads**.
2. Click on the branch/deployment you want to rebuild.
3. Click the **Retry deployment** (or **Deploy** button depending on status) to force a rebuild.

### 5.4 Rollback

Cloudflare Pages keeps a full deployment history, allowing instant rollbacks:

1. Go to [Cloudflare Dashboard](https://dash.cloudflare.com) → **Workers & Pages → fundbeads → Deployments**.
2. Find the last known-good deployment.
3. Click **⋯ → Rollback to this deployment**.

Alternatively, revert the offending commit on `release` and push to let Cloudflare redeploy.

## 6. Local Development

```bash
# Install dependencies
pnpm install

# Start dev server with hot reload
pnpm dev

# Build production bundle
pnpm build:frontend

# Build + run all tests
pnpm check
```

Dev server runs at `http://localhost:5173` by default.
